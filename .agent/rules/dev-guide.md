1. 文档先行：在制定任何计划之前，必须深度阅读相关的设计说明书。当收到“阅读文档”指令时，应通览项目目录，精准识别所有关联章节。
2. 严禁臆造 API：使用插件功能前必须查阅官方文档，严禁凭空捏造 API 接口。首次查阅新插件后，应及时将 URL 记录至“参考文档/Plugin Documentation Links.md”，并主动通过网络搜索获取准确信息。
3. 由博返约：先确立全局实现架构与策略，再详细拆解模块及其步骤，最后分步执行。
4. 架构至上：严禁由于追求进度而妥协整体架构设计；禁止为了短期的编码便利而牺牲系统未来的可扩展性。
5. 文档统一性：确保各文件间的信息准确、同步，严禁出现逻辑矛盾或内容冗余。
6. 及时重构拆分：当文档内容过长、职责过杂时，应立即进行物理拆分，并利用文件夹结构进行逻辑化管理。
7. 工具效能最大化：面对繁琐问题优先思考技术最优解，充分挖掘现有插件、外部工具及搜索引擎的潜力。
8. 拒绝重复造轮子：深度利用引擎与插件既有特性。在功能实现时，框架及插件功能应具有最高优先级。
9. 拒绝草稿代码：严禁为了验证功能而编写零散的临时 Demo；每一行代码都必须是最终系统实现的一部分。
10. 面向未来设计：所有系统设计必须预留扩展接口。不要因为项目尚处早期阶段而过度简化底层逻辑。
11. 进度实时存证：及时更新“待办文档/”目录下的文件，确保当前的开发进度与状态随时处于可追溯、可验证的状态。
12. 严格联网规范：始终遵循联网游戏开发范式，在编写任何逻辑时必须考虑全局上下文：数据同步、网络延迟与权限授权。
13. 术语定位精确：严格区分函数与事件；在描述业务流或逻辑跳转时，禁止使用含糊不清的词汇。
